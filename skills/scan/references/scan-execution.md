## Commit Conventions 分析 → `{{HARNESS_MEMORY_DIR}}/notes/rules/commit-conventions.md`

分析 git 历史与仓库配置，记录项目既有的提交、分支、PR、Tag 约定。下游 skill（Design / Worker）据此生成符合项目既有提交风格的产物。

### 1. 提交信息格式（commit message format）

运行 `git log --oneline -100` 并分析最近 100 条提交：

- **格式检测**：识别主导风格
  - Conventional Commits（`feat:` / `fix:` / `chore:` / `refactor:` / `docs:` 等）
  - Angular-style（带 scope：`feat(api):` / `fix(ui):`）
  - gitmoji（`:sparkles:` / `:bug:` / `:memo:` 前缀）
  - ticket-prefixed（`JIRA-123:` / `#42:`）
  - 自由格式（无固定前缀）
- **Subject 行长度**：在采样的 100 条提交上测得的 P95（用于推断 commitlint/husky 是否生效）
- **Body 使用率**：带 body 的提交占比（多行提交 / 单行提交）
- **Footer 模式**：识别尾部约定
  - `Signed-off-by:`（DCO 签署）
  - `Co-authored-by:`（多人协作）
  - `Breaking-Change:`（破坏性变更标识）
  - `Fixes #N` / `Closes #N`（issue 关联）

### 2. 分支命名约定（branch naming）

运行 `git branch -r` 并对远端分支抽样分析：

- **模式检测**：
  - `feature/<slug>` / `feat/<slug>`
  - `fix/<slug>` / `bugfix/<slug>` / `hotfix/<slug>`
  - `release/<version>`
  - `chore/<slug>` / `docs/<slug>`
  - 扁平命名（无前缀，仅 slug）
- **附 2-3 个具体示例**（取自远端真实分支名）

### 3. PR 模板（pull request conventions）

检查项目中是否存在 PR/MR 模板文件：

- `.github/pull_request_template.md`
- `.github/PULL_REQUEST_TEMPLATE.md`
- `.gitlab/merge_request_templates/*.md`
- `docs/pull_request_template.md`

**若存在**：仅记录文件路径，**不复述模板内容**（下游运行时直接读取模板）。
**若不存在**：标记为 "无模板"。

### 4. Changelog 格式

检查项目根目录的 `CHANGELOG.md` / `HISTORY.md` / `RELEASES.md`：

- **格式识别**：
  - [Keep a Changelog](https://keepachangelog.com/)（含 `## [Unreleased]` / `### Added/Changed/Fixed`）
  - 自动生成（如 `conventional-changelog` / `release-please` 标识）
  - 自定义格式（项目特有）
- **若存在**：记录格式名称与文件路径
- **若不存在**：标记为 "无 changelog"

### 5. Tags 与 Releases

运行 `git tag --sort=-creatordate | head -20`（有限抓取最近 20 个 tag）并分析命名：

- **命名模式**：
  - `v1.0.0`（带 `v` 前缀，SemVer）
  - `1.0.0`（裸 SemVer）
  - 基于日期（`2024.01.15` / `20240115`）
  - 其他（calendar versioning / 项目特有）
- **附 2-3 个具体示例**

## 规则补充

- **只读**：本步骤仅执行 `git log` / `git branch -r` / `git tag` 等只读命令，不修改任何 git 状态。
- **基于证据**：每项格式声明必须附 2-3 个真实示例（取自实际 git 历史）。
- **不复述外部模板内容**：PR 模板文件仅记录路径，下游运行时按需读取。
- **静态分析工具检测原则同步适用**：若发现 `commitlint.config.*` / `.commitlintrc*` / `husky` 配置，仅记录工具名 + 配置路径 + 运行命令（如 `npx commitlint --from HEAD~10`），**不打开**配置文件内容。
