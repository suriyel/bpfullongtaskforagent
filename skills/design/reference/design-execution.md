# design 节点执行规则（澄清回炉）

本文档承载 design 节点 audit 阶段识别的 4 项内部矛盾的处理规则。SKILL.md 主体定义流程框架，本文件定义矛盾点的具体决策语义。**主体与本文件冲突时以本文件为准。**

---

## § Step 4b 澄清 — 新建项目空表问询

**矛盾**：Step 4b.2 "通过 AskUserQuestion 询问用户补充内部库 / 静态分析工具 / 测试阈值" 与"design 阶段尽量减少与用户交互"的总体倾向冲突。

**处理规则**：

- **不强制问询**。若用户在前序对话（Step 1 / Step 1.6 读取的 user-original-intent.md 或对话上下文）中已提及具体内部库 / 静态分析工具 / 测试阈值 → 直接填入 §11 对应小节
- 若未提及 → **保留空表**（列头 + 0 行），作为后续 init / TDD 阶段的迭代输入，**不发起 AskUserQuestion**
- 留空 §11 在 Step 5b 不视为缺陷；空表本身即合法状态（标识"待迭代补全"）

**意图**：design 阶段 §11 的职责是"承载已知约定"，不是"挖掘新约定"；后者归 init / scan 节点。

---

## § Step 5b 档位适配 — 第 5 项硬精确条目传递的 L2/L3 放宽

**矛盾**：Step 5b 第 5 项要求"SRS §4 FR / §5 IFR 中每个 camelCase / snake_case 标识 / 文件路径 / 类型范围 / 默认值 / 枚举集合在 Design §6.2 中 1:1 出现"，但 L2/L3 纯参数配置场景下，SRS 可能列出大量衍生参数（如 derived ratios、computed bounds），强行 1:1 复现会导致 §6.2 冗长且与设计抽象层级不符。

**处理规则**：

- **核心参数 1:1**：SRS 中显式标记为"用户输入参数"或"配置入口参数"的硬精确条目，**必须** 1:1 出现在 Design §6.2.1（配置文件 schema）或 §6.2.2（任务书 / API schema）
- **衍生参数允许省略**：SRS 中从核心参数计算得出的衍生值（如 `searchRadiusMeters = baseRadius * scaleFactor`）允许在 §6.2 中**省略具体值**，但**必须标注计算来源**（如"衍生自 baseRadius / scaleFactor，详见实现"）
- **判定方法**：grep SRS 时区分"输入项"（参数表 / 配置 schema 节）与"派生项"（计算公式 / 推导段落）；仅输入项纳入 1:1 检查
- L1 输入下本项无放宽（L1 走全结构填写，§6.2 不会被压缩）

**意图**：保持下游实现阶段拿到的接口契约与 SRS 输入面 100% 对齐，同时避免把 SRS 的内部推导逻辑机械复制到 Design。

## § 复用映射 L2/L3 处理 — reuse_map 落地检查跳过条件

**矛盾**：Step 5b 第 6 项要求"reuse_map 每行对应 FR 需在 Design §4 出现 file:line 引用"，但 L2/L3 纯参数配置增量场景下 §4 整节常标 `[不适用]`，导致检查无落点。

**处理规则**：

- **§4 整节标 `[不适用]` 时，reuse_map 落地检查自动 PASS-SKIPPED**，附 skip 理由"§4 在 L2/L3 纯参数配置场景下不适用，reuse_map 落地点位移至 §6.2 配置 schema 引用"
- 若 §4 部分小节标 `[不适用]` 但整节存在 → 检查范围限定为存在的 §4.N 子节
- §4 整节 `[不适用]` 时，Step 2b 产出的 `reuse_map` 应转移到 §6.2 "复用说明" 备注列（若有），或直接登记到 §10 待解决问题"L2/L3 增量下的存量代码影响面"
- Step 2b PASS-SKIPPED（greenfield / 无复用面）→ 本项继续 PASS-SKIPPED（与原 SKILL.md 规则一致）

**意图**：检查项必须有可落地的检查面；L2/L3 场景下 §4 空洞时 reuse_map 仍需保留可追溯性，但落点从 §4 迁移到 §6.2 / §10。

## § 自检 vs 审阅门职责分离 — Step 5b 与 Step 5c 角色边界

**矛盾**：Step 5b "集成一致性检查" 与 Step 5c "自检与收尾" 都在做"检查"，角色容易混淆，可能导致 LLM 在 5b 阶段过度介入内容判断或在 5c 阶段忽略用户审批。

**处理规则**：

- **Step 5b 是 LLM 形式校验**：
  - 仅做**结构 / 格式 / 字段存在性**的机械检查（章节是否存在、表格行数是否 ≥ 1、字段是否 1:1 出现等）
  - **不做内容质量判断**（如"这个技术栈选择是否合理"、"这个决策替代理由是否充分"）
  - 输出格式：在草稿文件末尾 `## 待审阅事项` 章节追加检查结果，每项一个 `###` 子标题；0 标记时写"无标记"

- **Step 5c 是用户实质审批**（外部审阅门）：
  - **不修改设计内容**（设计修改属于审批前的对话迭代，不属于 5c）
  - 仅做两件事：
    1. 把首行 `**状态**: 待审批` 改为 `**状态**: 已批准`
    2. 删除文件末尾 `## 待审阅事项` 整段
  - 然后 `git commit`
  - **LLM 不自行决定通过 5c**——5c 触发条件是用户回复"批准"（在 Step 5 落盘后的告知文本之后）

- **失败路径**：
  - 5b 标记问题 → 在 `## 待审阅事项` 中留痕 → 用户在 5c 前可回 Step 4 修订（5b 自身不阻塞，仅提示）
  - 用户在 5c 不批准 → 回 Step 3 / Step 4 修订（LLM 不应自行通过）

**意图**：5b 是 LLM 自检层（形式机械），5c 是用户审批层（实质内容）。两层分工保证设计文档既经过结构校验，又有用户最终把关。
